OT Beyond the Primer: Control Algorithms and the TP2 Puzzle

The introductory account of Operational Transformation (OT) presents a tidy picture: define a transformation function, apply it pairwise, watch the replicas converge. The reality is more uncomfortable. The pairwise transformation function is only half of the story — equally important is the control algorithm that decides which ops to transform against which others, and in what order. Most published OT bugs over the past three decades have been in the control algorithm, not in the transformation function itself.

The TP2 puzzle

Recall that the transformation function must satisfy TP1 (two-op convergence) and TP2 (three-op convergence in a particular pattern). TP2 says, informally, that transforming an op through two concurrent ops should yield the same result regardless of the order in which the two ops are considered. Several influential OT algorithms — including the original adOPTed and GOTO algorithms — were later shown to violate TP2 on adversarial input. The Oster et al. paper "Real time group editors without Operational transformation" (2005) walks through specific counterexamples that motivated much of the subsequent CRDT research.

Control algorithm families

  • dOPT — the original algorithm; simple, easy to implement, but has known TP2 problems on three or more concurrent edits.
  • adOPTed — extends dOPT with explicit history buffers per site; correctness depends on TP2 holding.
  • GOTO / GOT — uses inclusive and exclusive transformations and tries to handle TP2 by careful history management.
  • SOCT2, SOCT4 — sidestep TP2 by enforcing a total order on operations via a sequencer, reducing the problem to TP1 only.
  • Jupiter / star-topology OT — assumes a central server that linearises all ops; each client only needs to transform against ops it has not yet acknowledged. This is the model behind Google Docs.

Why star-topology OT works in practice

The Jupiter model is popular for good reason: with a single server linearising operations, you avoid the multi-way concurrency that triggers TP2 entirely. Each client maintains an "outgoing buffer" of ops not yet acknowledged by the server, transforms incoming server ops against that buffer, and sends locally to the server. The server holds the canonical history and broadcasts transformed ops to other clients. Correctness collapses to TP1, which is tractable.

Operational complexity

The implementation difficulty of an OT system rarely lives in the transformation function — usually a few dozen lines per op type — but in maintaining the right history at the right level of fidelity to make transformations meaningful. Pruning history aggressively breaks reordering; keeping it forever exhausts memory. Production OT systems often retain a sliding window keyed on the slowest live connection.

When to reach for CRDTs instead

If your topology is genuinely decentralised — peer-to-peer, multi-master databases, occasionally-connected mobile devices — the engineering cost of building a correct OT control algorithm typically exceeds the cost of adopting a CRDT. OT remains preferable when you already have a central server and want the smallest possible per-op overhead.
