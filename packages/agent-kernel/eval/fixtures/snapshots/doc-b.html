Cache Invalidation Patterns

"There are only two hard things in computer science: cache invalidation and naming things." The joke is funny because it is, in practice, true. Once a cache holds a copy of state that may change at its source, you must decide how that copy is corrected — and every option has trade-offs that ripple through correctness, latency, and operational complexity.

Time-based invalidation

The simplest pattern: every cached entry carries a time-to-live (TTL), and once the TTL elapses the entry is considered expired. The cache discards or refetches on next access. TTLs are popular because they require zero coordination — the writer of the underlying data does not need to know the cache exists. The downside is that within the TTL window the cache may serve arbitrarily stale data, which is unacceptable for some workloads (account balances, permissions, feature flags affecting safety).

Event-driven invalidation

Here the system that mutates the source of truth publishes an event, and cache nodes subscribed to those events invalidate or update the affected entries. This trades coordination cost for freshness: caches can be near-instantly correct, but you must build, monitor, and reason about the event channel itself. Lost or delayed events become silent staleness bugs.

Write-through and write-behind

If the cache sits between the application and the database, the application's writes can flow through the cache. Write-through updates the cache and the source synchronously, so the cache is never stale relative to the source. Write-behind queues the write and applies it asynchronously, accepting a window of risk in exchange for write latency. Both eliminate the explicit invalidation problem at the cost of locking the application into one cache topology.

Versioning and content-addressable keys

A subtle and powerful pattern: bake the version into the cache key. If a user record's key is user:42:v17 rather than user:42, then mutating the user bumps the version to v18 and the old entry simply becomes unreachable rather than stale. Storage cost goes up — old versions linger until evicted — but invalidation becomes trivial. CDNs use this pattern heavily by including a content hash in asset URLs.

Negative caching

An often-forgotten cousin: caching the fact that a lookup returned nothing. This is invaluable for protecting an upstream system from repeated lookups for non-existent keys, but you must invalidate the negative cache when the missing entry is created — a common source of "I added it but the system says it doesn't exist" bugs.

Operational rule of thumb

Choose the weakest invalidation strategy your correctness requirements allow. TTL alone is often enough; reach for event-driven only when staleness has real cost. Combine strategies cautiously — every additional invalidation path is another way for the cache and the source of truth to disagree.
